data systems

notes on data, AI, and the shape of modern systems

I'm working through how modern data systems fit together. Not as a glossary of tools, not as a tutorial. More like a map. Every system in this space exists to solve a specific part of the same problem, and understanding the problem first makes every piece easier to place.

The sections below walk through each stage. Within each one, I try to explain what it is doing, why it exists, and where the interesting tradeoffs show up.

The tools change constantly. The four stages do not.

02
sources → storage → transformation → consumption → decision

sources

Before any analysis, any dashboard, any AI answer, something has to happen in the real world and get recorded somewhere. Sources are the origin point. They are the systems where data first exists.

source typewhat it isexamples
operational databases live databases behind products and services Postgres, MySQL backing a web app or SaaS product
event streams continuous records of actions as they happen user clicks, page views, server logs, IoT sensors
third-party platforms data from external systems via APIs or connectors Salesforce, Stripe, Google Ads, HubSpot
files and batch exports manual or scheduled data drops CSV exports, FTP transfers, flat files from vendors

This is where most of the messiness starts. Each source produces data in its own shape, at its own speed, with its own guarantees about reliability. A lot of data engineering is just dealing with that inconsistency before anything useful can happen downstream.

within sources

operational systems (OLTP)

The most common source is an operational database: the live system running a business or product. These are often called OLTP systems, short for online transaction processing.

OLTP systems are built for one thing: handling many small operations very fast, without making mistakes. When a customer places an order, a payment clears, or a record gets updated, that goes through an OLTP system. Latency matters here in a way it does not anywhere else in the stack. A slow checkout is a broken experience.

OLTP databases are optimized for writing and looking up individual records quickly, not for running large analytical queries across millions of rows. That is why data almost always has to move out of the operational system before any serious analysis can happen.

within sources

batch vs streaming

Data leaves its source in two fundamentally different modes, and the rest of the system has to be designed around whichever one you are working with.

batch

data is collected over a period of time and moved all at once. a nightly export, a file drop, a scheduled pipeline run. simpler to build, adds latency between when something happens and when the data is available.

streaming

data is processed as it arrives, event by event. lower latency, closer to real time. more complex to build and operate. tools like Kafka, Kinesis, and Flink sit in this layer.

Most organizations run both. Streaming for the things that need to be current. Batch for everything where a few hours of lag is fine.

The quality of everything downstream depends on what comes in here. A well-understood, clean source makes every stage after it easier. A messy or undocumented source creates problems that compound at every step.

sources → storage → transformation → consumption → decision

storage

Once data leaves a source, it has to live somewhere. Storage is where data is held at rest. It does not do anything by itself. It just holds data until something needs it.

The analogy that makes sense to me: storage is the boxes in a warehouse. The boxes hold the contents. They do not sort themselves, answer questions, or do any work. That is what compute is for.

storage

where data lives. boxes in a warehouse. holds records at rest. cheap to keep, designed to persist.

compute

what does the work. workers reading the boxes. reads data, runs queries, aggregates results. active while running, costs money while it runs.

Separating storage from compute is one of the ideas that makes modern cloud data architecture work. When they are decoupled, you can scale compute up when you need fast answers and scale it back down when you do not, without touching the stored data. Multiple teams can also run different compute engines against the same storage.

within storage

storage formats and where data lands

Not all storage is the same. The format data is stored in determines what kinds of queries are fast and what kinds are slow.

storage typewhat it isoptimized for
object storage flat files in a cloud bucket (S3, GCS, Azure Blob) cheap long-term storage, raw data lakes, parquet files
row-oriented databases records stored row by row transactional reads and writes, operational systems
columnar storage records stored column by column analytical queries that scan many rows but few columns
vector stores stores embeddings that represent meaning numerically semantic search, AI retrieval (RAG)

Columnar storage is significantly faster for analytical queries because you only read the columns a query touches. Row storage is faster for transactional work because you usually need a complete record all at once.

within storage

data warehouse, data lake, and lakehouse

Three terms that come up constantly, each representing a different answer to the question of where analytical data should live.

approachwhat it istradeoff
data warehouse a structured analytical database with managed storage and compute (Snowflake, BigQuery, Redshift) easy to query, governed, more expensive, less flexible with raw or semi-structured data
data lake raw files in object storage, often parquet or JSON, with compute run separately on top cheap and flexible, but querying it requires more engineering work and discipline
lakehouse combines lake storage with warehouse-style query capabilities (Delta Lake, Apache Iceberg) tries to get the best of both, still a more complex architecture to operate
within storage

analytical systems (OLAP)

OLAP stands for online analytical processing. Where OLTP (in sources) is optimized for writing many small records quickly, OLAP is optimized for reading large amounts of data to answer analytical questions.

This is where data warehouses live. When an analyst runs a query to calculate revenue by region for the last quarter, that is OLAP work. Large scans, aggregations across millions of rows, slower but deeper.

Most analytical data flows from OLTP to OLAP before it can be used. The OLTP system runs the business. The OLAP system helps understand it. They need very different things from the infrastructure underneath them.

within storage

types of databases

Not all databases are trying to do the same thing. The category a database sits in determines what it is fast at, what it is bad at, and where it belongs in the system.

categorywhat it optimizes forexamples
transactional (OLTP) fast individual writes and lookups. row-oriented. built to run applications. Postgres, MySQL, SQLite
cloud data warehouse large-scale analytical queries across many rows. managed infrastructure, scalable compute. Snowflake, BigQuery, Redshift
fast analytical (OLAP) extremely fast columnar queries, often at real-time or near-real-time scale. self-hosted or cloud. ClickHouse, Apache Druid, StarRocks
embedded / in-process runs inside your application or script, no server required. queries local files directly. DuckDB, SQLite
object storage cheap, durable file storage. not a database itself, but where raw data often lives before it gets queried. S3, GCS, Azure Blob (parquet, JSON, CSV files)
vector databases stores embeddings and retrieves semantically similar records. built for AI and search use cases. Pinecone, Weaviate, pgvector (Postgres extension)

The reason this landscape is fragmented is that each category makes a different tradeoff. A transactional database that is also fast at analytical queries would require compromises that make both worse. Most organizations end up using more than one type, moving data between them depending on what they need to do with it.

within storage

cloud offerings on top

Many of these database categories have a managed cloud version that sits on top of the core engine. The pattern is consistent: take something that is powerful but requires infrastructure to run, and wrap it in a service that handles the operational work for you.

base enginecloud offeringwhat the cloud layer adds
DuckDB MotherDuck shared access, persistence, collaboration, and cloud storage without giving up local speed
ClickHouse ClickHouse Cloud managed infrastructure, autoscaling, backups, and a web UI on top of the same fast engine
Postgres Neon, Supabase, AlloyDB serverless scaling, branching, built-in auth and APIs, removing the need to manage a database server
open table formats (Iceberg, Delta) Databricks, Snowflake, AWS Glue query engines and governance layers on top of files that live in your own object storage

What cloud offerings typically add across the board: managed infrastructure so you do not run servers, autoscaling so you are not paying for idle capacity, access controls and audit logging, web UIs for collaboration, and pay-as-you-go pricing. The core query engine stays the same. The operational burden disappears.

The tradeoff with managed cloud offerings is usually cost and control. You give up some flexibility and pay a premium over running it yourself, in exchange for not having to think about infrastructure. For most teams, that trade is worth it until the scale or specific requirements make self-hosting the better answer.

sources → storage → transformation → consumption → decision

transformation

Raw data from sources is almost never ready to answer a question directly. It is inconsistent across systems, missing context, and carries no shared definitions. Transformation is the stage where raw data gets shaped into something a person or system can actually trust.

This is also the stage with the most moving parts. It involves moving data between systems, modeling it, scheduling the pipelines that keep everything current, and building in the quality checks that make downstream consumers confident in what they are seeing.

within transformation

ingestion

Before any transformation can happen, data has to move from its source into the storage layer. That movement is called ingestion. It sounds simple but is where a lot of operational complexity lives: handling API rate limits, schema changes, deduplication, and keeping data fresh.

Tools like Fivetran, Airbyte, and Stitch are built to manage this. They connect to hundreds of source systems and land data into a warehouse or lake on a schedule, without requiring custom engineering for every connector.

within transformation

modeling

Once raw data is in storage, it needs to be structured into something reliable. Modeling is the process of building layered SQL transformations that take messy source tables and produce clean, documented, tested outputs.

raw source
exactly as it landed
→
staging
cleaned and renamed
→
intermediate
joined and enriched
→
mart
defined, tested, ready

The output of modeling is not just data. It is data with definitions attached: a revenue metric that says exactly what it counts, how it is calculated, and which tests confirm it is correct. dbt is the tool most commonly used for this work today.

Without modeling, every downstream consumer makes their own assumptions about what the numbers mean. Different dashboards produce different answers to the same question. Modeling is where those assumptions get written down and enforced in one place.

within transformation

orchestration

Transformation pipelines do not run once. They run on a schedule: nightly, hourly, or triggered by upstream events. Orchestration is the layer that manages the timing and order of all these jobs, handles failures, and makes sure dependencies run in the right sequence.

If ingestion has to complete before modeling can start, and modeling has to complete before a dashboard can refresh, something has to coordinate that chain. Tools like Apache Airflow, Dagster, and Prefect sit in this layer.

Bad orchestration shows up as stale dashboards, silent failures, and pipelines that succeed individually but produce wrong answers when combined.

within transformation

data quality and observability

Pipelines break. Schemas change upstream without warning. A column that was always populated suddenly has nulls. Data quality is about detecting these problems before they reach the people relying on the data.

This happens at two levels. Tests that run as part of the pipeline (checking expected row counts, uniqueness, referential integrity) and observability tools that monitor data health in production (Monte Carlo, Soda, Elementary). Together they answer: did this pipeline run correctly, and does the output look right?

within transformation

the semantic layer

Even after data is modeled and tested, there is one more problem: different teams define the same metric differently. Finance calculates revenue one way, sales calculates it another. When the numbers do not match, people stop trusting any of them.

The semantic layer is where metric definitions get locked in: what revenue means, what churn counts, how active users are calculated. Once defined there, any downstream tool, whether a dashboard, an analyst running a query, or an AI agent, gets the same answer.

dbt's Semantic Layer and MetricFlow sit in this space. They let you define a metric once and expose it consistently across every consumer.

within transformation

machine learning

Machine learning is mostly a transformation-stage concern on the input side. Before a model can be trained, data has to be cleaned, features have to be engineered, and training datasets have to be assembled from the same trusted models that power dashboards.

Feature engineering is where domain knowledge gets encoded into the inputs a model will learn from. Which signals matter, how they are normalized, what time windows to use: these decisions live here, and they are often the biggest driver of model quality.

Once a model is trained, it shifts stages. The trained model becomes a consumer and a producer: it takes data in, produces predictions, and those predictions often flow back into storage to be used downstream. That part shows up in the consumption stage.

consumption

Consumption is how data reaches a person or a downstream system. The transformation stage produces trusted, defined data. Consumption is everything that sits on top of it: the interfaces, the tools, and the systems that turn that data into answers.

There are a few different access patterns here, each suited to a different kind of question.

within consumption

BI and dashboards

The most familiar form of consumption. A BI tool (Tableau, Looker, Sigma, Metabase) connects to the storage and transformation layer, runs queries, and displays the results visually. An analyst or business user navigates to a dashboard, finds the number they need, and interprets it.

Dashboards are good at answering questions that were anticipated. Someone built the chart because they knew people would need that view. The limitation is that the data can only answer the questions someone thought to ask in advance.

within consumption

AI as the interface layer

AI sits between the transformation output and the person asking the question. Instead of a fixed dashboard, a person describes what they want in plain language and the model figures out which data to retrieve, how to interpret it, and what to say back.

Three patterns show up in how AI systems interact with data:

patternwhat it meanstradeoff
RAG before answering, the model looks up relevant documents or records from a data source more grounded answers using current or private data, more moving parts
tool calling the model calls a database, API, or calculator instead of guessing from memory more accurate and verifiable, more system complexity
agents the model loops through observe, decide, act until a multi-step task is complete handles complex tasks end to end, harder to predict and control

RAG is worth understanding on its own. An LLM has a knowledge cutoff and no access to your private data. RAG solves this by retrieving relevant records before the model generates its answer. Think of it as letting the model open the textbook before answering the question, rather than answering from memory.

AI does not change the underlying data problems. It changes the interface to the data. For an AI to answer well, the data still has to be clean, defined, and trustworthy. A confident wrong answer is worse than no answer, and that is exactly what happens when an AI agent hits undocumented or inconsistent data.

within consumption

ML models in production

Trained machine learning models are also a form of consumption. They take data from the storage and transformation layer, run inference, and produce predictions that flow somewhere else: a recommendation in a product, a churn score in a CRM, a fraud flag on a transaction.

This is often called model serving or inference. The model was trained offline in the transformation stage. In production, it consumes data continuously and produces outputs that other systems act on.

A key challenge here is model drift: the world changes, but the model does not update automatically. Monitoring model performance in production, retraining on fresh data, and managing versions of deployed models is its own discipline, sometimes called MLOps.

within consumption

data products and APIs

Not all data consumption is human-facing. Some of the most important consumers of a data platform are other systems: a recommendation engine reading a feature table, an application enriching user records with model outputs, a partner system receiving a data feed.

Data products treat trusted datasets as first-class outputs, with defined schemas, ownership, and SLAs. Data APIs expose that data programmatically, letting other systems query or subscribe to it without needing direct warehouse access.

sources → storage → transformation → consumption → decision

the decision

The whole system exists to produce a better decision than someone could make without it. That is the same goal from the beginning, and it is worth coming back to at the end.

For a decision to be trustworthy, the data behind it has to earn that trust at every stage.

propertywhat it means in practicewhere it gets established
accurate the numbers reflect what actually happened source quality, data quality testing
defined everyone using a metric agrees on what it counts modeling, semantic layer
timely the data is fresh enough to be relevant orchestration, streaming vs batch choices
accessible the right people or systems can reach it consumption layer, access controls
traceable you can follow a number back to where it came from lineage, documentation across every stage

The pattern I keep coming back to: tools change faster than the underlying questions do. Where does the data live? Where does compute happen? Who can trust the result? How does a human make a better decision because of it? Those questions were true before AI and they are still true now. The systems that answer them well are the ones worth understanding.
