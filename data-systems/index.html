what the data stack becomes

the infrastructure AC runs on

AC is the all-knowing intelligence from Asimov's 1956 story: a system that never guesses and is only limited by what it knows and how much it can process. Not a chatbot, not a copilot. Something that has complete context and never has to make up the difference. That is the target.

I wrote about two things that constrain intelligence: compute and context. The data problem AI can't solve explains what stands between today and AC. This is the infrastructure that has to exist for it to be possible. The data stack has always existed to produce better decisions. When the thing making those decisions is AC, the four stages don't go away. They become what it runs on. What they need to produce changes completely.

compute + context

Quantum computing is not far off in the abstract. IBM, Google, Microsoft Azure Quantum, IonQ, and a growing wave of startups are in labs right now. It doesn't have a commercial use case yet, but when it does, the unlock will be sudden. Healthcare, encryption, cryptography, AI: all of it changes when compute stops being the constraint.

That creates a forcing function for every layer of this stack. Compute and context move together. When compute takes a step-change, context has to match it or the additional processing power goes to waste. AC only gets smarter if both sides improve. The cycle of better compute demanding better context, better context enabling better compute to do more, is what drives toward AC. This page is about what each layer has to become for that cycle to keep moving.

The chain is the same. What sits at the end of it has changed. Each section below describes what the layer looks like in the AC era, with a note on what it replaced and why.

02
sources → storage → transformation → consumption → decision

sources

In the AC era, every source is a live signal. Not a file to import on a schedule, a feed AC reasons over continuously. Every piece of data gets a timestamp at the moment it enters the system. AC weights recency. It won't operate on context past its freshness threshold because stale context produces wrong answers, and wrong answers erode trust faster than no answer at all.

Coverage gaps are reasoning gaps. If a critical signal isn't connected, AC doesn't know what it doesn't have. It answers without flagging the absence. The competitive advantage shifts from having more data to having fewer holes in what you're giving the system.

mechanics

operational systems (OLTP)

The most common source is an operational database: the live system running a business or product. These are called OLTP systems, short for online transaction processing. They are optimized for one thing: handling many small operations very fast, without making mistakes. When a customer places an order, a payment clears, or a record gets updated, that goes through an OLTP system.

OLTP databases are not designed for analytical queries. Running heavy compute against them directly would degrade the product. In the AC architecture, the solution is CDC (change data capture), a stream of changes that leaves the source system mostly untouched while feeding a compute node that sits next to it.

mechanics

batch vs streaming

Data leaves its source in two fundamentally different modes.

batch

data is collected over a period of time and moved all at once. a nightly export, a file drop, a scheduled pipeline run. simpler to build, adds latency between when something happens and when the data is available.

streaming

data is processed as it arrives, event by event. lower latency, closer to real time. more complex to build and operate. tools like Kafka, Kinesis, and Flink sit in this layer.

In the AC era, streaming is the default. Batch still exists for cases where latency is genuinely acceptable, but those cases have to earn their place. The question is no longer "why would we stream?" but "what justifies the lag?"

before AC

Batch was the default. Nightly exports, hourly syncs, file drops: data moved on schedules because streaming required more engineering work and cost more to operate. The tradeoff made sense when the consumer was a dashboard that refreshed once a day. When the consumer is something reasoning in real time over current context, that tradeoff inverts. Streaming stops being an advanced pattern and becomes the baseline.

storage

In the AC era, storage is AC's working memory. Not a place where analysts run queries, but a layer AC navigates directly: structured tables to scan, vectors to search semantically, metadata to read like a map. Vector stores stop being a niche AI pattern and become core infrastructure.

The architecture is shifting from a central warehouse to a federation of domain-local compute nodes. Instead of moving data to a warehouse where compute runs against it, compute moves to where data already lives. Each source domain gets its own node that handles ingestion, transformation, and storage together, co-located with the source. The warehouse becomes the federation of those nodes.

storage

where data lives. holds records at rest. in the AC era, structured for machine navigation, not just human queries.

compute

what does the work. in the AC era, moving closer to the source, embedded in domain nodes rather than running in a central warehouse.

mechanics

storage formats

The format data is stored in determines what kinds of queries are fast and what kinds are slow.

storage typewhat it isoptimized for

row-oriented databases records stored row by row transactional reads and writes, operational systems
columnar storage records stored column by column analytical queries that scan many rows but few columns
vector stores stores embeddings that represent meaning numerically semantic search, AI retrieval (RAG)
mechanics

data warehouse, data lake, and lakehouse

approachwhat it istradeoff
data warehouse a structured analytical database with managed storage and compute (Snowflake, BigQuery, Redshift) easy to query, governed, more expensive, less flexible with raw or semi-structured data
data lake raw files in object storage, often parquet or JSON, with compute run separately on top cheap and flexible, but querying it requires more engineering work and discipline

before AC

The warehouse model (pull everything into Snowflake or BigQuery, run queries against it) made sense when the bottleneck was compute and you needed a central place to aggregate across domains. Separating storage from compute was the right call: scale compute when you need it, down when you don't. The next move reverses the direction. Instead of data flowing to compute, compute moves to data. Domain-local nodes sit close to source systems, receive a continuous stream, and handle transform and storage in place. The central warehouse becomes a federation. Storage cost was never the problem. It has been cheap for years. The problem was always structure: how do you organize what you have so an agent can navigate it fast?

04
sources → storage → transformation → consumption → decision

transformation

In the AC era, transformation is where the company's understanding of itself gets written down. The semantic layer is the contract AC reads: what revenue means, what churn counts, what an active user is. When AC reasons across your data, it uses your definitions, not guesses. dbt is where that contract lives.

Transformation is also continuous now, not scheduled. Definitions are enforced at write time, inline, as data enters the compute node. The pipeline and the query are the same operation. When quantum computing takes compute from a constraint to a commodity, context becomes the only real differentiator, and context lives here.

mechanics

ingestion

Before transformation can happen, data has to move from its source into the storage layer. That movement is ingestion, where a lot of operational complexity lives: API rate limits, schema changes, deduplication, freshness.

Tools like Fivetran, Airbyte, and Stitch are built to manage this. They connect to hundreds of source systems and land data into a warehouse or lake on a schedule, without requiring custom engineering for every connector. In the AC architecture, the ingestion layer gets absorbed into the domain compute node, but the problem it solves, reliably connecting to every source system, does not go away.

mechanics

modeling

Modeling is the process of building layered SQL transformations that take messy source tables and produce clean, documented, tested outputs.

The output of modeling is data with definitions attached: a revenue metric that says exactly what it counts, how it is calculated, and which tests confirm it is correct. dbt is the tool most commonly used for this work today.

mechanics

orchestration

Transformation pipelines run on a schedule: nightly, hourly, or triggered by upstream events. Orchestration manages the timing and order of all these jobs, handles failures, and makes sure dependencies run in the right sequence. Tools like Apache Airflow, Dagster, and Prefect sit in this layer.

In the AC architecture, orchestration shifts from schedule-based to event-based: compute triggers on a new stream event rather than a clock. The coordination problem doesn't go away; the cadence changes.

mechanics

data quality and observability

Pipelines break. Schemas change upstream without warning. A column that was always populated suddenly has nulls. Data quality is about detecting these problems before they reach the consumers relying on the data, whether that consumer is an analyst or AC.

Tests that run as part of the pipeline (checking row counts, uniqueness, referential integrity) and observability tools that monitor data health in production (Monte Carlo, Soda, Elementary). In the AC era, a data quality failure isn't just a broken dashboard. It is AC reasoning over wrong information and not knowing it.

mechanics

the semantic layer

The semantic layer is where metric definitions get locked in: what revenue means, what churn counts, how active users are calculated. Once defined there, any downstream consumer, whether a dashboard, an analyst running a query, or an AI agent, gets the same answer.

dbt's Semantic Layer and MetricFlow sit in this space. In the AC era, the semantic layer is not a query-time convenience. It is a write-time requirement. Definitions are enforced at ingestion, before anything gets stored, so that everything AC reads is already structured against the same contract.

before AC

Transformation ran on schedules: nightly dbt runs, hourly pipeline refreshes. The output was tables for analysts to query and dashboards to display. The semantic layer was a convenience, a way to keep metrics consistent across teams. In the AC era, the output is structured context for an agent to reason over, and the semantic layer is a hard requirement. Without it, AC guesses. Two companies with the same compute can get completely different answers from AC because one spent years making their transformation layer precise and the other didn't. You can't fix a missing or wrong definition after the fact.

consumption

In the AC era, the primary consumer at the end of the chain is an agent. AC doesn't browse a dashboard. It queries structured data, reads lineage, calls tools, and returns an answer. Consumption patterns built for human eyes have to be rebuilt for machine eyes.

The question shifts from "how do we display this data?" to "how do we make this data queryable by something that never sleeps and never guesses?" Dashboards don't disappear. They become one surface among many, and not the primary one anymore.

mechanics

BI and dashboards

A BI tool (Tableau, Looker, Sigma, Metabase) connects to the storage and transformation layer, runs queries, and displays the results visually. An analyst navigates to a dashboard, finds the number they need, and interprets it.

Dashboards answer questions that were anticipated. The limitation is that the data can only answer what someone thought to ask in advance. In the AC era, that limitation is exactly what AC resolves, but dashboards remain useful for human-facing reporting and executive visibility.

mechanics

mechanics

data products and APIs

Not all data consumption is human-facing. Data products treat trusted datasets as first-class outputs with defined schemas, ownership, and SLAs. Data APIs expose that data programmatically, letting other systems query or subscribe to it without needing direct warehouse access. In the AC architecture, these become the primary way domains expose their structured output to the rest of the federation.

before AC

The BI dashboard era was built for human consumption: navigate to a chart, find a number, interpret it, decide. That interface assumed the consumer had time to look, context to apply, and could handle some ambiguity in the data. The primary way information moved from the data stack to a decision was a visual display. In the AC era, the primary interface is a query. AC reaches directly into structured context without a human navigating a screen to translate what the data says.

the decision

The decision layer splits. AC computes: it reasons from the structured context below it, surfaces a recommendation, and can explain its reasoning. The human judges: whether to trust the recommendation, whether to act, what happens next.

The analyst who spent a career building dashboards becomes the strategist who interprets what AC surfaces and decides what to do about it. The role doesn't go away. It goes up.

propertywhat it means in practicewhere it gets established
accurate the numbers reflect what actually happened source quality, data quality testing
defined everyone using a metric agrees on what it counts modeling, semantic layer
timely the data is fresh enough to be relevant orchestration, streaming vs batch choices
accessible the right people or systems can reach it consumption layer, access controls
traceable you can follow a number back to where it came from lineage, documentation across every stage
before AC

Before AC, the decision layer was human all the way through. Data informed a person who made a judgment call. The quality of that judgment depended on how good the data was, how much context the person held in their head, and how well they could synthesize it under time pressure. AC doesn't replace that judgment. It improves the information going into it, and it never forgets context between questions. The human's job shifts from "find the insight" to "decide what to do with the insight AC already found."

Tools change faster than the underlying questions do. Where does the data live? Where does compute happen? Who can trust the result? How does a human make a better decision because of it? Those questions were true before AC. They are still true now. The stack described here is what it looks like when you take them seriously all the way through.

the shift

streaming and compute collapse at the source

In the AC architecture, there is no separate ingestion step. CDC streams from source systems directly into domain compute nodes that sit close to the source, one node per domain. The node receives the stream, runs transformations inline, enforces metric definitions at write time, and stores structured output in place. The distinction between moving data and computing on it dissolves.

The OLTP problem still exists: live databases running your product cannot have heavy analytical compute running against them directly. The solution is CDC: a stream of changes that leaves the source system mostly untouched. The compute node sits next to the source, not inside it. Source freshness is enforced by the architecture, not by tests that run afterward.

the shift

the warehouse becomes a federation

A central warehouse made sense when data had to travel to compute. In the AC architecture, compute is distributed: each source domain has a node, nodes produce structured output, and AC queries the federation. There is no single repository. There is a set of semantically consistent, well-defined outputs that AC can reason across together.

The "one massive cheap database that holds everything" path has surface appeal but the wrong bottleneck assumption: storage has been cheap for years. The constraint has always been structure. A federation of well-structured domain nodes, each governed by the same semantic contract, gives AC what it actually needs: not more storage, but better-organized context that it can navigate fast.

the shift

the context layer is the moat

Any organization can add compute nodes. Not every organization has spent years encoding what their data actually means. The semantic layer (documented metric definitions, tested models, traceable lineage) is what dbt becomes in this architecture: the governance contract that tells every node what its output is allowed to mean.

When quantum computing takes compute from a constraint to a commodity, this layer is the only real differentiator. Better compute is available to everyone. Better context belongs to whoever built it.

an open question

on quantum and local vs cloud compute

The "local quantum nodes that eventually become cheap enough for truly distributed compute" vision has real appeal. The more likely path is quantum following the GPU arc: powerful, then expensive, then democratized through cloud providers (AWS Braket, Azure Quantum, Google Quantum AI) rather than moving into local hardware. Local quantum nodes are probably 20 years further out.

The model is more likely "quantum as a service your federation calls out to when it hits a problem that needs it" than "quantum chips embedded in every compute node." What quantum changes is the ceiling: workloads that are currently impractical, like reasoning over an entire company's knowledge graph in real time or running thousands of hypothetical scenarios simultaneously, become practical. That is what unlocks the full version of AC.
